# /enh-implementation

ENH 설계 문서 작성부터 코드 구현, PR 생성까지 전체 워크플로우를 진행합니다.
내부적으로 `/enh-doc` 워크플로우를 사용하여 문서를 생성/갱신합니다.

## 사용법

```
/enh-implementation <ENH번호> "<제목>"
```

예시: `/enh-implementation 27 "비동기 이미지 로딩 개선"`

인자를 생략하면 현재 대화 컨텍스트에서 추론합니다.

---

## 실행 순서

### STEP 1 — 문서 초안 작성

`/enh-doc`의 실행 순서를 그대로 따릅니다:
- ENH 설계 문서 생성/갱신 (`devel-note/Enhancing/ENH-{번호}_*.md`)
- `devel-note/Enhancing/INDEX.md` 갱신

문서 작성 완료 후 작성된 내용을 요약하여 보여줍니다.

---

### STEP 2 — 승인 요청

`AskUserQuestion`으로 다음을 묻습니다:

> "ENH 문서 초안을 작성했습니다. 검토 후 아래 중 하나를 선택해주세요:
> 1. **승인** — 이대로 개발을 시작합니다
> 2. **수정 요청** — 수정할 내용을 알려주세요"

---

### STEP 3 — 수정 반복 (필요 시)

수정 요청이 있으면:
1. 지시한 내용에 따라 ENH 문서와 INDEX.md를 `/enh-doc` 워크플로우로 갱신합니다.
2. 변경된 내용을 요약하여 보여줍니다.
3. STEP 2로 돌아가 다시 승인을 요청합니다.

승인이 날 때까지 STEP 2~3을 반복합니다.

---

### STEP 4 — 코드 구현

승인 후 ENH 문서의 **작업 순서** 및 **영향 범위** 섹션을 기준으로 코드를 구현합니다.

- 작업 간 병렬 가능 여부를 판단하여 효율적으로 진행합니다.
- 구현 중 설계와 다른 점이 발견되면 사용자에게 보고 후 진행합니다.
- 구현 완료 후 변경된 파일 목록을 보고합니다.

---

### STEP 5 — 커밋 및 PR 생성

**커밋:**
- 브랜치: `claude/analyze-feature-tree-depth-fziak` (현재 작업 브랜치)
- 변경 파일 스테이징 후 커밋 생성
- 커밋 메시지 형식: `feat: implement ENH-{번호} {간략한 설명}`

**PR 생성:**
- GitHub MCP 도구(`mcp__github__create_pull_request`)로 PR 생성
- PR 제목: `[ENH-{번호}] {제목}`
- PR 본문: ENH 문서의 개요 및 작업 목록 요약 포함

PR URL을 최종 보고합니다.
