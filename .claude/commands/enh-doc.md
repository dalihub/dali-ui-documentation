# /enh-doc

ENH 설계 문서를 생성하거나 갱신하고, `devel-note/Enhancing/INDEX.md`를 동시에 업데이트합니다.

## 사용법

```
/enh-doc <ENH번호> "<제목>"
```

예시:
- `/enh-doc 27 "비동기 이미지 로딩 개선"` — 신규 생성
- `/enh-doc 26 "doc quality improvements"` — 기존 문서 갱신

인자를 생략하면 현재 대화 컨텍스트에서 ENH 번호와 제목을 추론합니다.

---

## 실행 순서

### 1. 파라미터 확인

- 인자로 ENH 번호와 제목을 받습니다.
- 인자가 없으면 현재 대화에서 가장 최근에 논의된 ENH 번호와 주제를 추론합니다.
- ENH 번호가 불명확하면 `AskUserQuestion`으로 확인합니다.

### 2. 기존 ENH 문서 존재 여부 확인

- `devel-note/Enhancing/` 폴더에서 `ENH-{번호}_*.md` 패턴 파일을 탐색합니다.
- **파일 없음 → 신규 생성** 모드
- **파일 있음 → 갱신** 모드 (기존 파일명 유지)

### 3. ENH 문서 작성/갱신

현재 대화 컨텍스트(논의 내용, 설계 결정사항, 코드 변경 계획)를 기반으로 작성합니다.

**파일 경로 규칙:**
```
devel-note/Enhancing/ENH-{번호}_{제목_소문자_언더스코어}.md
```

**신규 생성 시 문서 구조:**

```markdown
# [ENH-{번호}] {제목}

## 개요

{작업 목적 및 해결할 문제 1~3문장 요약}

---

## 작업 {N}: {작업명} (`{주요_파일명}`)

### 문제

{현재 상태와 문제점}

### 해결

{변경 내용 및 구현 방법. 코드 스니펫 포함}

### 영향 범위

| 파일 | 변경 내용 |
|------|-----------|
| `경로/파일명` | 변경 내용 |

---

## 전체 영향 범위 요약

| 파일 | 작업 |
|------|------|
| `경로/파일명` | 작업 번호 |

## 작업 순서

{병렬 가능 여부 포함한 작업 실행 순서}
```

**갱신 시:** 기존 구조를 유지하면서 변경된 내용만 수정합니다. 구현 상태가 완료된 경우 해당 섹션에 명시합니다.

### 4. INDEX.md 갱신

`devel-note/Enhancing/INDEX.md`를 업데이트합니다.

**신규 생성 시:**
- 적절한 단계(기존 패턴 참고)에 새 항목 행을 추가합니다.
- 해당 단계가 없으면 새 단계 섹션을 추가합니다.
- 하단 **전체 진행 현황 요약** 테이블도 갱신합니다.
- 구현 상태는 `🔴 미구현`으로 설정합니다.

**갱신 시:**
- 해당 항목의 주제 설명이나 구현 상태를 수정합니다.
- 완료된 경우 `✅ 완료`로 변경합니다.

### 5. 완료 보고

작성/갱신된 파일 목록과 주요 변경사항을 간략히 보고합니다.
